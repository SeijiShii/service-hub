## 1. 症状

ユーザー直接報告 (4 件):

1. **#1 (Low、UX 文言不足)**: admin form の「service-info endpoint」フィールドにユーザーが入力すべきは「フルパス (`https://example.com/api/hub/service-info`)」か「path のみ (`/api/hub/service-info`)」かが UI から分からない。
2. **#2 (Low、UX 文言不足)**: admin form の「サブドメイン」フィールドが**何のための値か不明**。URL とは別の意味かどうかも UI から分からない。
3. **#3 (High、致命機能不全)**: 既存サービスを「編集」して値を変更 → 「更新」ボタンを押しても**保存されない** (= ユーザー観測。実機 prod での操作)。
4. **#4 (Low、wording)**: actions 列の「退役」ボタン。日本語として不自然 (社内用語) → 「削除」が自然。

## 2. 期待挙動

1. endpoint フィールドに**入力形式が明示**されている (placeholder or help text)。
2. subdomain フィールドの**用途が明示**されている (or 用途が無いなら field 自体を撤去)。
3. 編集 → 更新 で値が DB に反映され、**ユーザーに「保存しました」フィードバックがある**。失敗時は「保存に失敗しました」+ 原因表示。
4. ボタン文言が「削除」(自然な日本語) になっている。

## §12 タイムラインから抽出した仮説

- **仮説 1 (#3 主因)**: submit() が `onSave(d)` を **fire-and-forget** で呼び、即 `setF({...empty})` + `setEditing(false)` でフォームをクリアしている。await が無い + saving spinner が無いため、user 視点では「ボタン押して何も起きなかった」印象になる。実態として PATCH は 200 で保存されているが、ユーザー目視に「保存された」確認がない。
- **仮説 2 (#3 副因)**: PATCH 失敗時 `setError("save_failed_<status>")` が Page top に表示されるが、form 即時クリア + table render との視覚干渉で error の発見が遅れる/見落とされる。
- **仮説 3 (#1)**: endpoint は full URL 要求 (providers adapter で `getJson(ref.endpoint)` 直接) だが、admin form の input には placeholder/help が無い。
- **仮説 4 (#2)**: subdomain は DB schema (`text("subdomain")`) と admin form に存在するが、**コード内のどこからも参照されていない dead field** (grep 結果: schema/queries/admin form のみ、providers/collection/dashboard 等の business logic から参照ゼロ)。記入意図が UI から不可解。
- **仮説 5 (#4)**: 「退役」は SPEC/D20260528-001 で `setServiceStatus → "retired"` から派生した button label。日本語として不自然なため「削除」へ変更すべき (DELETE handler が論理削除のため、「削除」でユーザー認知と一致する)。
