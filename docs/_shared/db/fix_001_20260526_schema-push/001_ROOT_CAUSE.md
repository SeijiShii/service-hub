# 根本原因分析: DB スキーマ適用機構の欠落

> **入力**: `./000_調査レポート.md`, `src/db/schema.ts`, `package.json`
> **最終更新**: 2026-05-26

## 1. 5 Whys
| # | 問い | 答え |
|---|---|---|
| Why 1 | なぜ API が DB クエリで失敗するか? | Neon にテーブルが存在しない |
| Why 2 | なぜ存在しないか? | スキーマを Neon に適用する手段が用意されていない |
| Why 3 | なぜ手段が無いか? | `drizzle.config` も `db:push`/`migrate` script も未整備 |
| Why 4 | なぜ未整備か? | 単体/結合テストは pglite（in-memory）で完結し、実 DB 適用は不要だった |
| Why 5 | なぜ完了扱いになったか? | **実 DB 適用は「実キーが要る release 工程」と暗黙に後回しされ、push 機構の整備自体が漏れた** |

**根本原因**: テストが pglite で自己完結したため、実 Neon へのスキーマ適用パス（config + script + 実行）が実装計画から漏れた。

## 2. 直接原因
| ファイル | 問題 |
|---|---|
| （不在）`drizzle.config.ts` | スキーマ適用のエントリポイントが無い |
| `package.json` scripts | `db:push` / `db:generate` が無い |

## 3. 根本原因
in-memory テスト戦略（pglite）と実 DB プロビジョニングの間にギャップ。実 DB への DDL 適用が誰の責務にも紐づいていなかった。

## 4. 寄与要因
| 種別 | 内容 |
|---|---|
| 運用ミス | マイグレーション/push の運用手順が未定義 |
| テスト | pglite テストは schema を直接 `client.exec` で流すため、実 push 経路を検証していない |

## 5. 仮説と検証
| 仮説 | 検証 | 結果 |
|---|---|---|
| Neon にテーブル無し | `pg_tables` クエリ 0 件 | ✅ |
| push 機構無し | config/script 探索 | ✅ |
